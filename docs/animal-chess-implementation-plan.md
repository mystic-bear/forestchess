# 동물 스킨 체스 학습 프로그램 구현 계획서

## 1. 문서 목적

이 문서는 기존 루미큐브 기반 웹앱을 "가족용 동물 스킨 체스 학습 프로그램"으로 전환하기 위한 구현 계획서다.

목표는 다음 네 가지를 개발 항목으로 분해하는 것이다.

- 정통 체스 규칙 100% 유지
- AI 1~7 난이도 제공
- 3단계 코치형 힌트 제공
- 기존 UI/워커 패턴 최대 재사용

이 문서는 기능명세서 다음 단계 문서로서 아래를 포함한다.

- 현재 코드 기준 전환 포인트
- 구현 로드맵
- 개발 체크리스트
- JIRA 스타일 에픽/티켓 분해
- Definition of Done

## 2. 현재 코드 기준 전환 전략

### 2-1. 유지할 영역

현재 코드에는 체스 전환에 바로 재사용할 수 있는 골격이 이미 있다.

- `js/ai-bridge.js`
  - worker 초기화, 요청 ID, timeout, partial 응답 캐싱, 오래된 응답 폐기 구조 재사용
- `js/game.js`
  - `stateVersion` 관리, 비동기 응답 무효화, `requestHint()` 흐름 재사용
- `js/ui.js`
  - 힌트 카드 렌더링 구조(`summary`, `leadText`, `reason`, `steps`) 재사용
- `worker/ai-worker.js`
  - `partialHint`, `hintResult`, `moveResult`, `error` 메시지 구조 재사용
- 세팅 화면 골격
  - 설정 리스트 렌더링, 빠른 시작 프리셋, 힌트 횟수 UI 패턴 재사용

주의

- 재사용 대상은 주로 구조와 비동기 패턴이다.
- `js/game.js`, `js/ui.js`, `shared/constants.js`, `index.html`은 루미큐브 전제 결합도가 높아 실제 코드 재사용률은 제한적일 수 있다.
- 따라서 공수 산정은 "구조 재사용" 기준으로 잡고, "파일 단위 재사용" 기준으로 잡지 않는다.

### 2-2. 교체할 영역

아래는 루미큐브 전용 로직이므로 체스용으로 교체해야 한다.

- `shared/rules-core.js`
  - 타일/그룹 규칙 전체 제거
- `worker/ai-base.js`
  - 루미큐브 수 생성, 탐색, 평가 로직 제거
- `worker/ai-dispatcher.js`
  - 루미큐브 레벨별 전략 분기 제거
- `worker/hint-engine.js`
  - 드로우/턴 종료/그룹 조작 중심 힌트 제거
- `js/game.js`
  - 4인 슬롯 모델, 손패/그룹/조커/등록 규칙 관련 상태 제거
- `js/ui.js`
  - 타일/그룹 렌더링 제거, 8x8 보드 렌더링으로 교체

### 2-4. 전환 원칙

- UI 껍데기와 비동기 파이프라인은 최대한 유지한다.
- 게임 코어와 AI 코어는 체스 기준으로 교체한다.
- 상대 AI와 힌트 AI를 분리한다.
- 힌트는 항상 최고 분석 기준으로 계산한다.

### 2-5. 선결정 운영 정책

개발 착수 전에 아래 정책을 확정한다.

#### 워커 구조

- 단일 워커 진입점은 유지한다.
- 공개 엔트리포인트는 계속 `worker/ai-worker.js`를 사용한다.
- `chooseMove`와 `getHint`는 같은 워커 계약을 유지하되, 내부에서 엔진 분석과 코치 분석을 분기한다.
- Stockfish 세션 관리와 코치 문장 생성은 워커 내부 보조 모듈로 분리한다.

이 결정의 이유는 다음과 같다.

- 현재 `js/ai-bridge.js`가 `worker/ai-worker.js` 계약에 깊게 맞춰져 있다.
- 진입 워커를 둘로 쪼개면 브리지, 요청 취소, partial 응답, timeout 정책을 동시에 다시 설계해야 한다.
- 체스 전환 초기에는 "단일 워커 진입점 + 내부 디스패치"가 가장 안전하다.

#### 힌트 세션 정책

- 한 포지션당 힌트 세션은 1회만 차감한다.
- 같은 포지션에서는 1단계 → 2단계 → 3단계로 확장해도 추가 차감하지 않는다.
- 같은 포지션에서 다시 힌트를 요청하면 기존 분석 캐시를 우선 재사용한다.
- 포지션이 바뀌면 새 힌트 세션을 시작한다.

#### 무승부 UX 정책

- 기물 부족: 자동 무승부 처리
- 스테일메이트: 자동 종료
- 3회 반복: 자동 감지 후 안내, 사용자 확인 버튼으로 종료
- 50수 룰: 자동 감지 후 안내, 사용자 확인 버튼으로 종료

#### 엔진 로딩 정책

- 엔진 파일 로드 실패 시 대국은 유지하고 AI/힌트만 비활성화한다.
- 초기 `ready`가 지연되면 재시도하고, 실패 원인을 사용자에게 짧게 노출한다.
- 분석 중에는 버튼 비활성화와 별개로 "분석 중" 상태를 화면에 표시한다.

## 4. 구현 로드맵

### Phase 0. 앱 전환 골격

목표: 루미큐브 앱을 체스 앱으로 부팅 가능하게 만든다.

작업 항목

- 4인 전제를 2인 White/Black 전제로 축소
- `SETUP_STATES`를 `HUMAN`, `AI-1` ~ `AI-7` 중심으로 교체
- quick start 프리셋을 체스 모드 기준으로 재구성
- 세팅 UI 문구를 체스 기준으로 교체
- 빈 체스판 화면 진입 가능 상태까지 정리

완료 기준

- 시작 화면에서 체스 세팅 화면으로 이동 가능
- White/Black 각각 Human/AI 설정 가능
- AI-1~AI-7 선택 가능
- 체스 대국 화면이 비정상 오류 없이 열림

### Phase 1. 체스 코어 MVP

목표: 정통 체스 규칙으로 사람 vs 사람 대국이 끝까지 가능해야 한다.

작업 항목

- 보드 상태 모델 구현
- 합법 수 생성 구현
- 체크/체크메이트/스테일메이트 판정
- 캐슬링/앙파상/프로모션 구현
- 3회 반복, 50수 룰, 기물 부족 무승부 처리
- FEN 생성/복원
- SAN 기록 및 move history 저장
- perft 기준 테스트 구축
- 특수 규칙 포지션 회귀 테스트 구축
- FEN 복원/재직렬화 일치 테스트 구축

완료 기준

- 사람 vs 사람 로컬 대국 가능
- 특수 규칙 포함 전 수순 정상 진행
- 종료 판정 정확
- perft 기준값 테스트 통과
- FEN round-trip 테스트 통과

### Phase 2. 보드/UI

목표: 가족용 학습 앱에 맞는 체스 화면을 만든다.

작업 항목

- 8x8 보드 렌더러
- 기물 선택 및 이동 인터랙션
- 합법 수 하이라이트
- 마지막 수 강조
- 체크 상태 표시
- 잡힌 기물 영역
- 수순 패널
- 프로모션 모달
- 동물 스킨/이름 병기 모드

완료 기준

- 마우스/터치로 대국 가능
- 초보자가 현재 턴, 체크, 마지막 수를 식별 가능
- 동물명/체스명 표시 모드 전환 가능

### Phase 3. Stockfish 연동

목표: AI와 둘 수 있는 체스 프로그램으로 만든다.

작업 항목

- Stockfish worker 연결
- UCI 초기화(`uci`, `isready`, `setoption`, `ucinewgame`)

- 분석 중 화면 상태 표시

완료 기준

- 사람 vs AI 가능

- 오래된 엔진 응답이 UI를 덮지 않음
- AI-1~AI-7 체감 차이 존재
- 엔진 로딩 실패 시 앱 전체는 유지

### Phase 4. Coach 힌트 시스템

목표: 이 제품의 핵심인 설명형 힌트를 구현한다.

작업 항목

- Coach 전용 분석 프로필 분리
- 3단계 힌트 구현
- partial hint 표시
- 대안 수 1~2개 표시
- 위협 설명 및 다음 그림 요약
- 보드 하이라이트 연동
- 실패 시 fallback 카드 제공
- 챌린지 모드 힌트 정책 차등 적용
- 포지션 단위 힌트 세션 캐시 및 차감 정책 구현

완료 기준

- 1단계: 방향 힌트
- 2단계: 후보 기물 힌트
- 3단계: 정답 힌트
- 엔진 실패 시 앱은 멈추지 않음
- 힌트는 불법 수를 추천하지 않음
- 같은 포지션에서 단계 확장 시 추가 차감이 발생하지 않음

### Phase 5. 저장/복기/학습 보조

목표: 반복 학습 가능한 제품으로 마감한다.

- 리뷰 모드
- FEN copy

완료 기준

- 앱 재실행 후 이어하기 가능

- 복기 데이터 재생 가능

## 9. Definition of Done

### 9-1. 티켓 단위 DoD

각 티켓은 아래를 만족해야 완료로 본다.

- 기능이 명세대로 동작한다.
- 정상 흐름과 예외 흐름을 모두 확인했다.
- 수동 검증 절차 또는 테스트가 남아 있다.
- 기존 기능을 깨지 않는다.
- 가족용 문장 톤을 해치지 않는다.

### 9-2. 프로젝트 단위 DoD

- 사람 vs 사람, 사람 vs AI가 끝까지 가능하다.
- 정통 체스 규칙 전체가 정상 동작한다.
- AI-1~AI-7 체감 차이가 존재한다.
- 3단계 힌트가 단계적으로 정보량을 늘린다.
- 힌트가 불법 수를 추천하지 않는다.
- 엔진 실패 시 앱이 멈추지 않는다.
- 자동 저장/이어하기가 가능하다.
- 종료 후 간단 복기가 가능하다.

## 10. 기술 리스크와 대응

### 10-1. 리스크: 규칙 코어보다 UI를 먼저 바꾸는 경우

문제

- 체스판은 보여도 규칙 오류가 누적될 수 있다.

대응

- 보드 UI보다 `ChessState`와 `move-generator`를 먼저 안정화한다.

### 10-2. 리스크: AI 강도와 힌트 강도를 같은 엔진 프로필로 묶는 경우

문제

- 낮은 AI와 두는 사용자가 낮은 품질 힌트를 받게 된다.

대응

- 상대 AI와 Coach 분석 프로필을 분리한다.

### 10-3. 리스크: 힌트를 최선 수 1개만 단정적으로 보여주는 경우

문제

- 박빙 포지션에서 신뢰가 떨어진다.

대응

- `confidence`, `alternativeMoves`, `MultiPV` 기반 문구를 사용한다.

### 10-4. 리스크: 워커 응답 경합

문제

- 이전 포지션 분석 결과가 현재 화면에 잘못 적용될 수 있다.

대응

- 기존 `stateVersion` 폐기 로직을 그대로 유지한다.

## 11. 현재 코드와 바로 연결되는 파일 목록

착수 전 가장 먼저 확인할 파일은 다음과 같다.

- `shared/constants.js`
  - 현재 `SETUP_STATES`, `AI_LEVEL_INFO`를 체스용으로 교체해야 한다.
- `js/game.js`
  - 현재 `quickStartMixedLevel`, `hintLimit`, `stateVersion`, `requestHint()` 흐름을 체스 기준으로 재사용 가능하다.
- `js/ui.js`
  - 현재 힌트 패널 렌더링 구조는 유지하고 보드/세팅 UI만 교체하면 된다.
- `js/ai-bridge.js`
  - 단일 워커 진입점 전제에서 worker 요청/partial/timeout/재시작 구조를 그대로 살릴 수 있다.
- `worker/ai-worker.js`
  - 체스 버전에서도 공개 엔트리포인트로 유지한다.
  - `partialHint`, `hintResult` 메시지 계약은 재사용 가치가 높다.
- `worker/hint-engine.js`
  - 반환 스키마는 참고하되 내용은 체스용으로 전면 교체가 필요하다.
